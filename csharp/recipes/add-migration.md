# Recipe: Add Migration

Use this when a feature changes the database schema and the change must roll out safely across mixed-version application instances.

Default tool is **EF Core migrations**: generated from the model diff, committed as C# alongside the snapshot, rendered to reviewable SQL, and applied by an explicit step — never automatically on normal startup. Choosing a SQL-first tool (DbUp, Flyway) instead is an ADR-level decision; see [../decisions/framework-selection.md](../decisions/framework-selection.md) and [../decisions/architecture-decision-records.md](../decisions/architecture-decision-records.md).

## Files To Touch

- `src/Orders.Infrastructure/Data/OrdersDbContext.cs` and `Data/Configurations/*.cs` — the model change that drives the migration
- `src/Orders.Infrastructure/Data/Migrations/<timestamp>_<Desc>.cs` (+ `.Designer.cs`) and `OrdersDbContextModelSnapshot.cs` — generated by the tool, committed, reviewed as code
- the apply seam — the `--migrate` startup flag handler or the init-job entrypoint that calls `Database.MigrateAsync()` — only if the apply strategy changes
- repositories and callers under `src/Orders.Infrastructure/Data` and any `Orders.Core` port that consumes the new shape
- integration tests under `tests/Orders.IntegrationTests` that run against a real database

## Steps

1. Change the model first (entity type plus its `IEntityTypeConfiguration<T>`), then generate: `dotnet ef migrations add <Desc> --project src/Orders.Infrastructure --startup-project src/Orders.Api --output-dir Data/Migrations`. Migration order is append-only: never edit, rename, or reorder a migration that has shipped — fix forward with a new one.
2. Read the generated `Up` **and** `Down` as reviewable code, not tool output. The `Down` must restore the prior schema, not just `DROP` the new objects blindly; hand-edit it (data-preserving reversals, index recreation) where the scaffold falls short.
3. Render the SQL and review it in the PR: `dotnet ef migrations script <From> <To> --project src/Orders.Infrastructure --startup-project src/Orders.Api` (add `--idempotent` for the deploy artifact). The C# diff is not the review surface — the SQL is what runs in prod.
4. Classify the change:
   - **Additive / safe** — nullable or defaulted `ADD COLUMN`, `CREATE TABLE`, new index, additive enum values. Old code ignores the addition; ship in one release.
   - **Destructive** — `DROP COLUMN`, `DROP TABLE`, `RENAME`, narrowing a type, adding `NOT NULL` to existing data, or any change that breaks code still running during rollout. These are **never** a single-release migration.
5. For every destructive change, do EXPAND/CONTRACT across releases so mixed-version instances stay correct:
   - **Release N (expand):** add the new column/table/index. Application code writes **both** old and new shapes and reads the old shape.
   - **Backfill:** populate the new shape from the old (a migration step or a one-off job), idempotent and re-runnable.
   - **Release N+1 (switch):** flip reads to the new shape; keep writing both until every instance is on N+1.
   - **Release N+2 (contract):** stop writing the old shape, then drop it. This `DROP` is the only destructive migration, and it ships only after no running version references the old shape.
6. Apply through the explicit step so dev, CI, and prod run identical migrations. Pick one and document it:
   - **Deploy-step apply (default for prod):** an init job or deploy stage runs the host with `--migrate` (which calls `MigrateAsync()` and exits) before the new app version receives traffic.
   - **Migrate-on-startup:** acceptable for single-writer/dev only; guard against concurrent appliers and never let startup apply a destructive migration silently. Normal startup never migrates — see [../services/database.md](../services/database.md).
7. Document the rollback story in the PR: which `dotnet ef database update <PreviousMigration>` is safe, which is not (a contracted `DROP` is forward-only — recovery is restore-from-backup, not down-migration), and the operator workflow for each.

## Invariants To Preserve

- Every migration is deploy-safe: either purely additive, or reversible, or staged as expand/contract.
- No destructive change ships in the same release as code that still depends on the old shape.
- Migration order is immutable once shipped — append a new migration, never edit history.
- `Up` and `Down` live together; `Down` actually restores the prior schema.
- The committed model snapshot matches the model — no pending model changes outside a migration.
- The change is tested against a real PostgreSQL, not the EF in-memory provider or SQLite standing in for a Postgres feature.

## Proof

- `dotnet ef migrations has-pending-model-changes --project src/Orders.Infrastructure --startup-project src/Orders.Api` reports none — the snapshot is in sync.
- Apply, roll back, re-apply against a real database (Testcontainers PostgreSQL, per [../quality/testing.md](../quality/testing.md)): `database update`, then `database update <PreviousMigration>`, then `database update` again — proving forward, reverse, and re-apply all succeed.
- `dotnet test tests/Orders.IntegrationTests --filter Repository` behind the `-Integration` switch — repository tests pass against the freshly migrated schema.
- The `dotnet ef migrations script` output is attached to or summarized in the PR and reviewed.
- For an expand/contract change, a test or documented review proving release N and N+1 application code both run correctly against the release-N schema.
- run `pwsh ./verify.ps1` — green.
