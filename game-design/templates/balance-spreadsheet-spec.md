# Balance Spreadsheet Spec: <PROJECT_NAME>

One-line purpose: the committed numeric model for <PROJECT_NAME> — every cost, reward, drop rate, timer, and difficulty scalar, with the derivation that justifies it.

## Purpose And Scope

- **Workbook location:** <URL OR REPO PATH>. Access: <WHO CAN EDIT / WHO CAN VIEW>.
- **Owner:** <PERSON OR ROLE>. Changes land by <EDIT DIRECTLY | REVIEWED COPY | PR ON EXPORTED FILES>.
- **In scope:** every player-facing tunable number — object costs and benefits, resource faucet and sink rates, XP and progression thresholds, drop tables, timers, difficulty scalars, DDA floor/ceiling bounds. If a designer would ever want to retune it, it has a row here.
- **Out of scope:** logic, content, and narrative. The live-tuning surface is this numeric layer only (see game-design/operations/live-tuning-and-telemetry.md).
- **Export path:** workbook -> <EXPORT FORMAT: csv/json/scriptable data> at <BUILD PATH OR REMOTE-CONFIG NAMESPACE>, generated by <SCRIPT OR MANUAL EXPORT STEP + WHO RUNS IT>. Hand-editing an exported file is a defect; fix the workbook and re-export.

## Tab Layout

One tab per concern; no numbers live outside these tabs. No merged cells, no color-as-data, no blank spacer rows — every tab must survive a machine export.

| Tab | Owns | Feeds |
|---|---|---|
| `Objects` | one row per balanceable object: all cost and benefit terms | `Curves`, `Export` |
| `Resources` | one row per resource/currency: faucets, sinks, intended net flow per session, terminal player value | `Simulation`, `Export` |
| `Curves` | fitted cost-curve parameters, one named fit per resource domain; exchange-rate assumptions | `Objects` deviation columns |
| `DifficultyCurve` | one row per level/chapter: skill tested, target fail rate, target completion time, planned rest valleys, measured values | difficulty tuning (game-design/recipes/tune-a-difficulty-curve.md) |
| `Simulation` | engagement profiles, session counts, per-profile drift outputs, target bands | economy sign-off (game-design/recipes/balance-an-economy.md) |
| `Export` | flattened `key -> value` view of everything that ships | build data / remote config |
| `ChangeLog` | append-only history of every value change | review and rollback |

- Add project-specific tabs (`DropTables`, `Matchups`, <OTHERS>) only with a row in this table and columns specified below in the same change.
- `Matchups` (if the game has intransitive counter relationships): one row per option pair with intended win rate at equal skill; the counter-cycle rules live in game-design/disciplines/difficulty-and-balance.md.

## Cost And Benefit Columns

`Objects` tab. Every cost and benefit is its own column with a stated unit — never a prose blob. Hidden terms are the usual source of "mystery" imbalance (Ian Schreiber, Game Balance Concepts Level 3, `gamebalanceconcepts.wordpress.com/2010/07/21/level-3-transitive-mechanics-and-cost-curves/`).

| Column | Type | Rule |
|---|---|---|
| `id` | string | stable, matches the build's data id; never reused |
| `name` | string | display name, informational only |
| `domain` | enum | the resource domain whose curve prices this object (one curve per domain) |
| `cost_<term>` (one column per term) | number + unit | resource price, drawback magnitude, setup requirement, opportunity cost — every term the player gives up |
| `benefit_<term>` (one column per term) | number + unit | every term the player gains |
| `cost_total` | formula | weighted sum of `cost_*` — weights live in `Curves`, never inline |
| `benefit_total` | formula | weighted sum of `benefit_*` |
| `curve_benefit` | formula | benefit predicted by the domain's fitted curve at `cost_total` |
| `deviation` | formula | `benefit_total - curve_benefit`; computed, never typed |
| `deviation_note` | text | required when `abs(deviation) > <THRESHOLD>`: why this object deliberately sits off the curve, with DDR link if it moves an invariant |
| `derivation` | text/link | where the raw terms came from: playtest, sim run, comparable object, or DDR |

- Adding a new cost or benefit term means a new column plus a weight row in `Curves` — updating every existing object's row in the same change, not just the new object's.
- `Resources` tab columns: `resource`, `faucets` (each with rate per session), `sinks` (each with rate per session), `net_flow_target` (band, not a point), `terminal_value` (the player desire it ultimately serves). Required per game-design/disciplines/economy-and-progression.md; a faucet with no listed sink fails review.

## Curve Formulas

- One fitted curve per resource domain, stored as named parameters in `Curves` (e.g. `curve_<domain>_a`, `curve_<domain>_b` for `benefit = a * cost^b`), so `curve_benefit` formulas reference names, not cell coordinates. Current fits: <LIST DOMAIN -> FORMULA + PARAMETERS + DATE FITTED>.
- Fit the late game separately. Benefits outpace costs at the top end in transitive systems (Schreiber's CCG observation — see game-design/disciplines/difficulty-and-balance.md); record the crossover point where the late-game fit takes over: <COST VALUE OR TIER>.
- Cross-currency comparisons require an explicit exchange-rate row in `Curves`: `<CURRENCY_A> = <RATE> * <CURRENCY_B>`, with the DDR that set it. No formula may convert currencies with an inline constant.
- Refitting a curve is a design decision, not maintenance: it reprices every object in the domain. Record it via game-design/decisions/design-decision-records.md and log it in `ChangeLog` with the old and new parameters.
- A reviewer must be able to recompute any object's `deviation` from its row plus `Curves` alone. If recomputation needs a hidden constant, the spec is broken — fix it.

## Simulation Hooks

- **Model:** <SPREADSHEET FLOW TAB | MACHINATIONS MODEL AT URL + VERSION>. If external, pin the model version next to the workbook and keep its inputs mirrored in `Simulation` so results are reviewable without the tool (Adams & Dormans, *Game Mechanics: Advanced Game Design*, ch. 5).
- **Profiles:** three fixed engagement profiles — low, median, high — defined as <SESSIONS/WEEK, MINUTES/SESSION, ACTIVITY MIX> each. Change a profile definition only via `ChangeLog` entry; drift history is incomparable across silent profile edits.
- **Run:** every faucet or sink change re-runs the simulation for <N> sessions per profile before merge (owned by game-design/disciplines/economy-and-progression.md). Record per profile, per resource: net currency drift, and whether it lands inside `net_flow_target`.
- **Outputs live in the workbook:** `Simulation` holds the latest run's date, inputs hash or version, and per-profile drift table. A sim result quoted in review but absent from the tab does not count as proof.
- **DDA bounds:** if the project ships any dynamic difficulty adjuster, its floor and ceiling values are rows here (`dda_<param>_floor`, `dda_<param>_ceiling`), per the bounded-adjuster rule in game-design/disciplines/difficulty-and-balance.md.

## Change Log Discipline

`ChangeLog` is append-only. Every value change — including reverts and corrections — is a new row. Never edit or delete history.

| Column | Rule |
|---|---|
| `date` | ISO date of the change |
| `author` | who changed it |
| `keys` | export keys or object ids touched |
| `old -> new` | exact values, both sides |
| `predicted_drift` | sim result for economy-touching changes; `n/a` with reason otherwise |
| `rationale` | the evidence: playtest finding, telemetry query, curve deviation, or DDR link — "felt right" fails review |
| `rollout` | `<SEGMENT NAME>` first, then global; live values ship segment-first per game-design/operations/live-tuning-and-telemetry.md |

- A change that moves a stated invariant (net-flow target band, curve parameters, exchange rate, DDA bounds, profile definitions) additionally requires a DDR; the row links it.
- The workbook, the exported config, and this spec move in the same change — a build value that disagrees with its workbook row is the defect flagged by game-design/checklists/design-review.md.
- Handoff rule: every value's `derivation` must let a new owner rebuild the number without asking anyone (see game-design/onboarding-and-handoff.md). Restore missing derivations before transferring ownership.
